Détail de la réalisation

Contexte et objectif Dans le cadre d'un projet de lab personnel en 1ère année, j'ai mis en place une infrastructure complète de déploiement automatisé de postes Windows, reproduisant les pratiques utilisées en entreprise pour la gestion de parc informatique. L'objectif était de pouvoir déployer un poste Windows configuré et joint au domaine en quelques minutes, sans intervention manuelle sur chaque machine. Infrastructure mise en place L'environnement repose sur Microsoft Hyper-V avec trois types de machines virtuelles : Un serveur FOG sous Debian 12 (génération 1, pour éviter les conflits EFI) hébergeant les services DHCP, TFTP, NFS et l'interface web de gestion Un contrôleur de domaine Windows Server avec Active Directory DS et DNS intégré, domaine gigafog.local Un poste master Windows 10/11 servant de modèle de référence pour les déploiements Le réseau a été soigneusement planifié avec des adresses IP fixes pour les serveurs (AD : 192.168.1.106, FOG : 192.168.1.120) et une plage DHCP pour les postes clients, avec distribution automatique du DNS et du suffixe de domaine via les options DHCP de FOG. Préparation du poste master La préparation du poste master a nécessité plusieurs étapes préalables avant toute capture : Désactivation de BitLocker pour permettre à FOG de monter la partition NTFS en lecture Désactivation de l'hibernation et du démarrage rapide (powercfg /hibernate off) pour éviter l'état "unclean filesystem" au moment de la capture Suppression du package MicrosoftOfficeHub qui bloquait systématiquement le processus Sysprep Installation du FOG Client (agent léger) permettant la communication entre le poste déployé et le serveur FOG pour l'exécution automatique des tâches post-déploiement Création d'un fichier de réponse unattend.xml pour bypasser l'écran OOBE (Out-Of-Box Experience) au premier démarrage, permettant à l'utilisateur d'arriver directement sur l'écran de connexion Windows sans passer par les étapes de configuration initiale Sysprep (/generalize /oobe /shutdown) pour généraliser l'image (nouveau SID unique pour chaque clone, suppression des informations spécifiques à la machine) Configuration du serveur FOG Côté serveur, plusieurs configurations ont été nécessaires : DHCP : configuration des options domain-name-servers et domain-name pour que chaque machine déployée reçoive automatiquement le bon serveur DNS et le suffixe de domaine sans manipulation manuelle NFS : création des dossiers /images et /images/dev avec les fichiers .mntcheck nécessaires au montage par FOG lors des captures/déploiements Active Directory Defaults : configuration dans l'interface FOG des identifiants de jonction au domaine (nom de domaine, compte Administrateur, mot de passe) appliqués automatiquement à chaque déploiement Processus de capture et déploiement La capture de l'image s'effectue via boot PXE : le poste master (éteint après Sysprep) démarre sur le réseau, FOG le reconnaît par son adresse MAC et lance automatiquement la capture en mode Multiple Partition Image - Single Disk, préservant toutes les partitions Windows (EFI, MSR, système, récupération). Le déploiement sur une nouvelle VM suit le même principe : création d'une VM vide sans OS, boot PXE, enregistrement dans FOG via Quick Registration, assignation de l'image et configuration de la jonction AD, puis lancement du déploiement. En fin de processus, le FOG Client exécute automatiquement la jonction au domaine gigafog.local et redémarre la machine — qui est alors immédiatement utilisable avec un compte de domaine. Difficultés rencontrées et solutions DifficultéSolution apportéeErreur Exec format error au boot PXEPassage de snponly.efi à ipxe.efi dans la config DHCPBitLocker bloquant la captureDésactivation complète avant SysprepSysprep bloqué par MicrosoftOfficeHubSuppression du package via PowerShell avant SysprepDHCP FOG distribuant le mauvais DNSAjout manuel des options domain-name-servers dans dhcpd.confConflit d'IP entre serveur AD et FOGAttribution d'IPs fixes distinctes sur chaque serveurFOG Client déclenchant un redémarrage intempestifDésactivation temporaire de la tâche AD join sur le host masterPartition NTFS en état "unclean"Désactivation du démarrage rapide et extinction propre avant capture Résultat La chaîne de déploiement est pleinement fonctionnelle : à partir d'une VM vide, le déploiement complet (boot PXE → image → jonction domaine → prêt à l'emploi) s'effectue en environ 15 à 20 minutes sans aucune intervention manuelle, contre plusieurs heures pour une installation Windows traditionnelle poste par poste.

Détail de la réalisation

Contexte et objectif Dans le cadre d'un projet personnel, j'ai déployé une solution de messagerie collaborative auto-hébergée en remplacement d'outils cloud comme Slack ou Teams, dans une logique de souveraineté des données et de maîtrise de l'infrastructure. L'objectif était de mettre en place un service de communication interne opérationnel, administrable et indépendant de tout service tiers. Infrastructure mise en place Le déploiement repose sur une VM dédiée (Debian/Ubuntu) hébergeant l'ensemble de la stack via Docker Compose, composée de deux conteneurs principaux : Rocket.Chat : l'application de messagerie web MongoDB : la base de données utilisée par Rocket.Chat pour stocker les messages, utilisateurs et configurations L'utilisation de Docker Compose permet de gérer les deux services de manière cohérente (démarrage, arrêt, mise à jour) via un fichier docker-compose.yml unique, avec des volumes persistants pour garantir la conservation des données même en cas de redémarrage ou de mise à jour des conteneurs. Déploiement via Docker Compose Le fichier docker-compose.yml définit les deux services avec leurs dépendances, variables d'environnement et volumes : yamlversion: '3' services: rocketchat: image: rocket.chat:latest restart: always environment: - MONGO_URL=mongodb://mongo:27017/rocketchat - ROOT_URL=http://192.168.1.x:3000 ports: - "3000:3000" depends_on: - mongo mongo: image: mongo:5.0 restart: always volumes: - ./data/db:/data/db Le démarrage s'effectue avec : bashdocker compose up -d Configuration de Rocket.Chat Une fois l'instance démarrée et accessible via le navigateur, plusieurs éléments ont été configurés : Gestion des utilisateurs et rôles : Création d'un compte administrateur principal Création de comptes utilisateurs supplémentaires pour les tests Attribution des rôles : admin, user, moderator selon les besoins Configuration des permissions par rôle (qui peut créer des canaux, inviter des membres, supprimer des messages, etc.) Configuration générale : Personnalisation du nom de l'instance et du logo Paramétrage des options d'inscription (inscription ouverte ou sur invitation uniquement) Configuration des notifications Difficultés rencontrées et solutions DifficultéSolution apportéeVersion MongoDB incompatible avec Rocket.ChatUtilisation de MongoDB 5.0 spécifiquement requis par la version de Rocket.ChatPersistance des données après redémarrageAjout de volumes Docker montés sur le système hôteAccès à l'interface d'administration bloquéReconfiguration du compte admin via les variables d'environnement au premier démarrage Résultat L'instance Rocket.Chat est pleinement opérationnelle et accessible depuis le réseau local. Les utilisateurs peuvent s'authentifier, communiquer via des messages directs et des canaux, avec une administration centralisée des comptes et des permissions.
